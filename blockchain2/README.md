1. Overview
This formative assessment extends the Blockchain-Based Attendance System built in the Practical Project done in formative 1. In the Practical Project, students built a system that records attendance as immutable blocks on a chain. In this formative, students will extend that system so that attendance marking automatically generates token transactions, and those transactions must be confirmed through mining before they are added to the chain. Students will implement both the UTXO model and the account-based balance model to manage student token balances, and simulate solo, pool, and cloud mining to confirm attendance blocks. The aim is to show how blockchain transactions and mining work together inside a real system.

2. How the Extension Works
In the formative 1, blocks were appended to the chain immediately after marking attendance. In this formative 2, that changes:

When a student is marked PRESENT, the system automatically creates a token reward transaction of 10 coins credited to that student
When a student is marked LATE, the system creates a token reward transaction of 5 coins
When a student is marked ABSENT, no transaction is created
The attendance block is not added to the chain immediately. It enters a pending pool
A miner must mine the pending pool to confirm the blocks and add them to the chain
Only after mining is the attendance record confirmed and the student balance updated
This means attendance marking, transactions, and mining are all connected in one flow.

Updated Block Structure

Students must add the following two fields to their existing block structure from the Practical Project:

A token reward field storing the coins earned: 10 for PRESENT, 5 for LATE, 0 for ABSENT
A transaction ID field storing the SHA-256 hash of the reward transaction
All other fields from the Practical Project remain unchanged.

3. Assessment Tasks

Part 1: Transaction (Ledger) Models
Students must implement two models for managing student token balances. Both models must use the student IDs from the existing registry as account identifiers. Each student starts with a balance of 0 and earns tokens only through confirmed attendance.

3.1 UTXO-Based Transaction Model
The program should:

Use transactions with inputs and outputs
Track unspent transaction outputs (UTXOs)
Prevent double spending
Calculate student balances from available UTXOs
Focus Areas:

UTXO data structures
Transaction validation
Balance calculation
Each transaction must deduct a fixed fee (e.g., 1 coin) from the total token amount before crediting the student. If a student spends tokens and inputs exceed the output amount plus fee, the excess must be returned to the student as a change UTXO. Transactions where inputs are insufficient to cover outputs plus fee must be rejected. The program must display the full UTXO set after each confirmed block, showing all unspent outputs and their owners.

3.2 Account-Balance Transaction Model
The program should:

Store each student as an account with a token balance
Update balances directly after each confirmed attendance block
Reject any manual token transfer where the sender balance is insufficient
Focus Areas:

Account balance management
Debit and credit operations
Each student account must include a nonce that increments with every outgoing transaction. Transactions submitted with an incorrect or reused nonce must be rejected. The program must maintain a transaction history log per student account stored as a linked list in memory, recording sender, recipient, amount, fee, and nonce. A command must be available to print the full transaction history for any given student ID.

Model Selection

The program should allow users to choose between:

UTXO model
Account-based model
This can be done using a menu, configuration option, or compile-time setting. The chosen model applies to all token transactions throughout the session.

Part 2: Mining Simulation
Mining is now required to confirm attendance blocks. Students must simulate three mining approaches. All three must work with the same pending pool of attendance blocks generated by marking attendance.

4.1 Individual (Solo) Mining
One miner validates all pending attendance blocks and creates a confirmed block
The miner receives the full mining reward on top of any student token rewards
Each block hash must begin with a configurable number of leading zero characters (difficulty level, default: 2). The program must increment a nonce field in the block and recompute the SHA-256 hash until the target is met. The program must display the number of hash attempts required before a valid hash was found. Difficulty must be adjustable via a command-line argument or menu option (range: 1 to 4).

 

 

4.2 Pool Mining
Multiple miners work together to confirm the pending attendance blocks
Mining rewards are shared among miners based on contribution

A pool fee of 2% is deducted from total rewards before distribution
Each simulated miner is assigned a random hash rate (attempts per round). Reward share is calculated as: (miner_attempts / total_attempts) * block_reward. Results must be displayed in a formatted table showing miner ID, attempts, share percentage, and reward received.

4.3 Cloud Mining
Users rent mining power to confirm pending attendance blocks instead of mining directly
Users pay a rental fee per round
Mining rewards are credited over time minus maintenance fees
Users must specify a rental duration (1 to 5 rounds). Each round, a fixed rental fee is deducted and a mining reward is credited. The program must calculate and display gross earnings, total fees paid, and net profit across all rounds. If cumulative fees exceed cumulative rewards at any round, the program must display a warning that the rental is unprofitable.
